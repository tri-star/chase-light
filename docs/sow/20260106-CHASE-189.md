# SOW: CHASE-189: データソース種別のコンポーネント化

## プロジェクト概要

**課題ID**: CHASE-189
**作成日**: 2026-01-06
**種別**: リファクタリング / 実装改善

## 1. 背景と目的

### 背景

現在、`/dashboard`、`/activities`、`/data-sources`など複数のページで「アクティビティ種別（Release/PR/Issue）」を表現するUIが存在するが、それぞれのコンポーネント内で個別に実装されており、以下の問題がある：

1. **重複コード**: 同じ`activityTypeLabels`と`activityTypeClasses`が4つのファイルで重複定義されている
2. **表記揺れ**: ラベル表記が「リリース」「Release」のように統一されていない
3. **保守性の低さ**: 色やスタイルを変更する際に複数ファイルを修正する必要がある

**現状の重複箇所：**

| ファイル | リリース表記 | Issue表記 | PR表記 |
|---------|------------|----------|--------|
| `ActivityListItem.vue` | リリース | Issue | PR |
| `ActivityDetailHeader.vue` | リリース | Issue | PR |
| `DataSourceActivityList.vue` | Release | Issue | PR |
| `NotificationCard.vue` | リリース | Issue | PR |

### 目的

アクティビティ種別を表示するBadgeコンポーネントを共通化し、以下を実現する：

1. 種別ごとの背景色・文字色を統一
2. ラベル表記を統一（日本語: リリース/Issue/PR）
3. 単一のコンポーネントで一元管理

## 3. 技術仕様

### コンポーネント設計

#### コンポーネント名

`ClActivityTypeBadge`

#### 配置先

課題で提案されている2つの選択肢について：

1. `packages/frontend/components/common/data-sources/` - 複数機能で再利用される中粒度UI
2. `packages/frontend/features/data-sources/components/` - 機能固有のコンポーネント

**選定: `packages/frontend/components/common/`**

理由：
- アクティビティ種別はダッシュボード、アクティビティ一覧、データソース詳細など複数の機能で使用される
- フォルダ構成ガイドラインで「複数機能で再利用される中粒度UI」は`components/common/`に配置と定義されている
- `base/`は広く共通の最小UI（`Cl*`接頭辞）であり、このコンポーネントもその性質を持つが、ドメイン固有のロジック（アクティビティ種別のラベル・色定義）を含むため`common/`が適切

#### Props仕様

```typescript
interface Props {
  activityType: 'release' | 'issue' | 'pull_request'
  size?: 'sm' | 'md' // デフォルト: 'md'
}
```

#### 内部実装

```typescript
// アクティビティ種別のラベル定義
const activityTypeLabels: Record<ActivityType, string> = {
  release: 'リリース',
  issue: 'Issue',
  pull_request: 'PR',
}

// アクティビティ種別のスタイル定義
const activityTypeClasses: Record<ActivityType, string> = {
  release: 'bg-status-info-subtle text-status-info-default',
  issue: 'bg-status-warn-subtle text-status-warn-default',
  pull_request: 'bg-status-success-subtle text-status-success-default',
}
```

### スタイル設計

既存の実装から統一されたスタイルを採用：

| 種別 | 背景色 | 文字色 | ラベル |
|-----|-------|--------|--------|
| release | `bg-status-info-subtle` | `text-status-info-default` | リリース |
| issue | `bg-status-warn-subtle` | `text-status-warn-default` | Issue |
| pull_request | `bg-status-success-subtle` | `text-status-success-default` | PR |

### サイズバリアント

| サイズ | パディング | フォントサイズ |
|-------|-----------|--------------|
| sm | `px-2 py-0.5` | `text-xs` |
| md | `px-3 py-1` | `text-xs` |

## 5. テスト戦略

### 既存テストでのカバー

共通コンポーネント単体のテストは作成せず、既存のページテストでカバーする方針とする。

理由：
- 共通コンポーネントの役割は表示のみであり、ロジックは最小限
- 既存のページテストで各ページでのBadge表示が検証されている
- テスト戦略で「APIルートはBFFとして薄い整形が多いため、ページ/機能テストを優先」という方針と整合

## 8. リスク・考慮事項

### 技術的リスク

- **既存テストへの影響**: コンポーネント構造の変更により、既存のテストセレクタが動作しなくなる可能性
- **スタイルの差異**: 既存実装間でパディングや角丸に若干の差異がある可能性

### 軽減策

- 置き換え後に既存テストを実行し、失敗がないことを確認
- 既存の最も標準的なスタイル（`ActivityListItem.vue`）をベースに統一
- Storybookでの視覚的確認を実施
